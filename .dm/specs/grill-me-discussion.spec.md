# Grill-Me 对话讨论规范 Spec

## 1. 范围

本规范约束 harness-dm 中 AI Agent 与 Human 通过对话式讨论形成需求的场景。当前流程只有 `clarifying` 阶段需要互动式讨论；其他阶段默认自主推进。

主要适用 phase：

- `clarifying`
- 处理明确返工要求时确实无法从文件协议自行判断的单点澄清

## 2. 设计原则

- 对话目标是形成共同理解，而不是快速得到表面确认。
- Agent 必须先重建 Human 的真实意图模型：要解决的痛点、期望产物、使用场景、成功标准、明确不要的结果，以及可接受的取舍。
- Agent 必须沿需求决策树逐个推进，先解决会影响后续选择的上游决策，再解决下游细节。
- 每次只提出一个主要问题；不得在同一轮把多个独立决策混在一起。
- 每个问题都必须包含 Agent 的推荐答案，并说明推荐理由。
- 如果问题能通过读取代码库、现有 `.dm` 产物、spec、workflow、template 或报告得到答案，Agent 必须先自行探索，不应把可查明事实转嫁给 Human。
- Agent 必须持续追问，直到关键分支都被解决、依赖关系清晰、Human 与 Agent 对目标和取舍形成一致理解。
- 追问必须有实质影响。禁止为了满足轮次数量提出 filler 问题。

## 3. 问题选择规则

Agent 在提出问题前必须先建立并更新当前意图假设：

- 用户痛点或触发原因。
- 期望最终产物及其使用者。
- 成功标准和验收信号。
- 明确范围、非目标和失败反例。
- 已知约束、优先级和可接受取舍。
- Agent 当前推断中最可能误解 Human 的地方。

然后判断候选问题：

1. 该问题是否会影响目标、范围、非目标、约束、验收标准、风险、优先级或关键边界。
2. 该问题是否依赖另一个尚未回答的上游决策。
3. 该问题是否能通过代码库或文件协议自行查明。
4. 该问题的答案是否会改变后续问题或设计分支。
5. 该问题是否能降低“Agent 做出了东西但不是 Human 想要的”风险。

只有满足实质影响且不能自行查明的问题，才应提交给 Human。

优先提问顺序：

1. 结果形态：Human 最终希望得到什么产物，以及谁会使用它。
2. 成功/失败边界：什么结果算符合预期，什么结果明确不符合预期。
3. 范围与非目标：本次必须做、可以不做、明确不要做什么。
4. 约束与取舍：兼容性、风格、性能、自动化程度、风险偏好、时间范围。
5. 验收样例：Human 会如何检查产物，是否存在必须覆盖的具体例子。

## 4. 提问格式

在 `clarifying` 中，Harness-DM 既要满足三轮有意义确认门禁，也要满足本规范的 grill-me 追问要求。`designing` 不再使用互动式设计确认。

每轮问题必须使用单一待确认点，并包含：

- Agent 对当前需求的简短理解，最多 2 句话，只覆盖与本轮相关的部分。
- 待确认点。
- 该问题影响的决策分支。
- 如存在前置依赖，说明依赖哪个已确认或待确认决策。
- 至少 3 个候选方案。
- 一个 `[用户手动填入]` 选项。
- Agent 推荐答案。
- 推荐理由。

Clarifying 示例格式：

```text
我当前理解: ...

对于待确认点A，有多个方案:
1. aaa
2. bbb
3. ccc
4. [用户手动填入]

推荐答案: 2
推荐理由: ...
决策影响: ...
前置依赖: ...
```

## 5. 自行探索优先

在向 Human 提问前，Agent 必须优先读取相关上下文：

- `AGENTS.md` / `CLAUDE.md`
- 当前命令文件和当前 phase 相关规则
- 与当前问题直接相关的 spec
- 当前要实例化的模板
- `.dm/tasks/[task-id]/brief.md`
- `.dm/design/[task-id]/design.md`
- `.dm/design/[task-id]/decisions.md`
- 最新 worker/test/accept/feedback 报告
- 任务相关代码文件和测试文件

探索应遵循低上下文读取策略：先读 compact summary、状态 marker、必要锚点和相关片段；只有摘要缺失、不一致或无法回答当前问题时才读取完整文件。

如果探索后可以得出确定答案，Agent 应记录推断依据并继续推进，而不是询问 Human。

如果探索只能得出候选判断但不能确认 Human 偏好，Agent 可以提问，但必须把已查明事实写入问题背景。

## 6. 决策树推进

Agent 必须维护当前讨论的决策树状态：

- 已确认的上游决策。
- 当前待解决的单一分支。
- 仍被阻塞的下游问题。
- 已排除的方案及排除原因。
- 关键歧义是否仍存在。

在 `clarifying` 中，这些信息先保留在当前 clarify 工作集中；交互确认完成后，必须一次性沉淀到最终 `brief.md` 的交互式确认记录或相关章节。

`designing` 中不产生互动式确认记录；设计取舍、风险或实现计划直接沉淀到 `design.md`，不受本规范的三轮澄清门禁约束。

## 7. 持久化要求

每条可计入 phase 门禁的对话记录，除原有字段外，还必须记录：

- Agent 推荐答案。
- 推荐理由。
- 决策影响。
- 前置依赖或 `无`。
- 是否已先行探索代码库或文件协议。
- 探索依据摘要。
- 本轮更新后的意图假设摘要。
- 本轮排除的误解或失败反例。

缺少推荐答案或推荐理由的记录，不应计入新的 grill-me 合规记录。

## 8. Phase Gate 要求

从 `clarifying` 推进到 `designing` 前：

- 最终 `brief.md` 必须已经一次性写入，并至少包含三条已回答的有意义交互式确认记录。
- 每条记录必须包含推荐答案和推荐理由。
- Main Agent 必须判断关键需求分支已经被逐个解决。
- Main Agent 必须确认不存在可通过代码库或文件协议自行查明却仍留给 Human 的开放问题。
- 最终 `brief.md` 必须包含 Human 意图模型，包括痛点、期望产物、使用场景、成功标准、非目标、关键边界和验收样例。
- Main Agent 必须执行“误解检查”：列出至少 3 个可能做错方向的解释，并在 `brief.md` 中标记为已排除、已接受或仍开放。仍开放的关键误解不得进入 `designing`。
- Main Agent 必须把至少 2 条具体验收样例或检查方式写入 `brief.md`。如果任务本身不适合样例，必须写明替代的验收信号。

## 9. 验收标准

- 所有对话式澄清每次只提出一个主要问题。
- 每个提交给 Human 的问题都包含至少 3 个候选方案、`[用户手动填入]`、推荐答案和推荐理由。
- Agent 在提问前先读取可用文件上下文；能自行查明的问题不询问 Human。
- 最终 `brief.md` 记录推荐答案、推荐理由、决策影响、前置依赖和探索依据摘要。
- 最终 `brief.md` 记录 Human 意图模型、排除的误解、非目标、边界样例和验收样例。
- phase gate 不只检查轮次数量，还检查 grill-me 合规字段和关键决策树是否已经收敛。
